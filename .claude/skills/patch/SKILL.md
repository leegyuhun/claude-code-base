# 코드 패치 스킬

## 수정 원칙

### 최소 범위 수정
필요한 코드만 변경한다. 버그와 직접 관련 없는 코드는 건드리지 않는다. 들여쓰기, 공백, 변수명 정리 등 스타일 수정은 하지 않는다.

### 읽기 우선
수정 전 반드시 전체 파일을 읽는다. 부분 컨텍스트로 수정하면 숨겨진 의존성을 놓칠 수 있다.

### 주석 처리 우선 삭제
로직을 제거할 때는 삭제보다 주석 처리를 먼저 고려한다. YSR은 레거시 시스템으로 숨겨진 의존성이 많다.

## 수정 워크플로우

`_workspace/01_investigation.md`의 수정 대상을 먼저 확인하고 아래 기준으로 경로를 선택한다.

### 경로 판단

| 조건 | 경로 |
|------|------|
| 수정 파일 2개 이하 **AND** 논리 단위 1개 (명확한 단순 수정) | **간단 수정** |
| 수정 파일 3개 이상 **OR** 논리 단위 2개 이상 | **스프린트 수정** |
| 공유 유닛(`ComUnit/`, `Common/` 등) 변경 포함 시 | **무조건 스프린트 수정** |

---

### 간단 수정 경로

계획서 작성과 [PAUSE]를 생략하고 바로 수정한다.

1. 대상 파일 읽기
2. 수정 실행 (CP949 규칙 준수)
3. 검증 (CP949 읽기 검증 / AssemblyInfo 버전 확인 등)
4. `_workspace/02_patch_summary.md` 작성

범위 외 발견사항이 있으면 `_workspace/04_out_of_scope.md`에 기록하고 건너뛴다.

---

### 스프린트 수정 경로 (3단계)

### Phase A: 수정 계획

`_workspace/01_investigation.md`를 읽고, 코드를 한 줄도 수정하기 전에 수정 계획서를 작성한다.

#### 논리 단위(Unit) 분할 원칙

- **의존성 있는 파일은 하나의 Unit**: `.pas` + `.dfm` 쌍, C# 프로젝트 파일 + `AssemblyInfo.cs`
- **무조건 1파일씩이 아님**: 관련 파일 묶음이 하나의 Unit. 단순 수정은 Unit 1개도 무방.
- **공유 유닛 변경은 별도 Unit**: `ComUnit/`, `Common/` 등은 독립 Unit으로 분리하여 위험 관리
- **Unit 수는 보통 2~4개**: 유지보수 수정은 대규모가 아니므로 과도한 분할 불필요

#### 수정 계획서 형식

`_workspace/01.5_patch_plan.md`에 저장:

```markdown
# 수정 계획서 — #{이슈번호}

## 수정 전략
[근본 원인에 대한 수정 접근 방식 1~2문장]

## 논리 단위

### Unit 1: {단위명} (예: "프린터 포트 갱신 로직 수정")
- 대상 파일:
  - `path/to/File1.pas` — 변경 내용 요약
  - `path/to/File1.dfm` — 변경 내용 요약 (있는 경우)
- 수정 방식: [구체적 접근법]
- 검증 방법: [CP949 읽기 검증 / 호출부 확인 등]
- 위험 요소: [공유 유닛 여부, 숨겨진 의존성 등]

### Unit 2: {단위명} (예: "AssemblyInfo.cs 버전업")
- 대상 파일:
  - `path/to/AssemblyInfo.cs` — 버전 +1
- 수정 방식: [마지막 자리 +1]
- 검증 방법: [버전 번호 확인]
- 위험 요소: [없음]

## 수정 순서
[Unit 간 의존성이 있으면 순서 명시. 없으면 "의존성 없음"]

## 범위 외 제외 사항
[조사 중 발견했지만 이번 수정에서 제외하는 항목. 없으면 "없음"]
```

#### [PAUSE] 체크포인트

계획서 작성 완료 후 아래 형식으로 출력하고 사용자 확인을 기다린다:

```
┌─────────────────────────────────────────┐
│ 수정 계획 — #{이슈번호}                  │
│                                         │
│ 수정 단위:                              │
│  Unit 1: {단위명} ({파일 수}개 파일)     │
│  Unit 2: {단위명} ({파일 수}개 파일)     │
│                                         │
│ 총 수정 파일: N개                        │
│ 공유 유닛 변경: 있음/없음               │
│                                         │
│ '실행' 입력 시 수정을 시작합니다         │
│ 계획 수정이 필요하면 알려주세요          │
└─────────────────────────────────────────┘
```

---

### Phase B: 점진적 수정

사용자가 '실행'을 입력하면 계획서의 Unit 순서대로 수정한다.

#### Unit별 수정 절차 (건너뛰기 금지)

각 Unit마다 반드시 아래 5단계를 수행한다:

1. **시작 선언**: `"--- Unit {N}: {단위명} 시작 ---"` 출력
2. **파일 읽기**: Unit의 모든 대상 파일을 먼저 읽는다
3. **수정 실행**: 계획서의 수정 방식대로 수정 (CP949 규칙 준수)
4. **단위 검증**: 계획서의 검증 방법 실행
   - `.pas`/`.dfm` 수정 시: CP949 읽기 검증 필수
   - C# 수정 시: AssemblyInfo.cs 버전 확인
   - 함수 시그니처 변경 시: 호출부 Grep 확인
5. **완료 선언**: `"--- Unit {N}: {단위명} 완료 ---"` 출력

#### 범위 외 발견사항

수정 중 계획 범위 밖의 문제를 발견하면 `_workspace/04_out_of_scope.md`에 기록하고 건너뛴다. 범위 확대는 하지 않는다.

```markdown
# 범위 외 발견사항 — #{이슈번호}

| 발견 항목 | 파일 | 설명 | 권장 조치 |
|-----------|------|------|-----------|
| ...       | ...  | ...  | 별도 이슈 등록 권장 |
```

#### 검증 실패 시

- 1차 실패: 원인 분석 후 재시도
- 2차 실패: `git checkout -- {파일}` 복구 후 다른 접근법 시도
- 3차 실패: 해당 Unit을 실패로 기록하고 다음 Unit으로 진행

---

### Phase C: 최종 확인

모든 Unit 완료 후 전체 결과를 정리한다.

1. **결과 요약 출력**:
```
┌─────────────────────────────────────────┐
│ 수정 완료 — #{이슈번호}                  │
│                                         │
│  Unit 1: {단위명}  ✅ 완료              │

│ 수정 파일: N개 / 계획: M개              │

└─────────────────────────────────────────┘
```

2. **`_workspace/02_patch_summary.md` 작성** (기존 형식 유지)
3. **`_workspace/04_out_of_scope.md` 작성** (발견사항 있을 때만)

## CP949 인코딩 보호 (HIGHEST PRIORITY)

`.pas`/`.dfm` 파일은 CP949 인코딩이다. **Edit/Write 도구 사용 절대 금지.**
모든 수정은 반드시 Python 스크립트 + Bash 도구로만 수행한다.

### 수정 절차 (3단계 필수)

**1단계: 인코딩 확인**
```bash
python -c "data=open('파일.pas','rb').read(3); print(data.hex())"
# efbbbf → UTF-8 BOM (Edit 도구 사용 가능)
# 그 외  → CP949 (Python 스크립트만 사용)
```

**2단계: Python 스크립트로 수정**
```python
# _workspace/fix_xxx.py 파일로 작성 후 Bash로 실행
with open('파일.pas', 'r', encoding='cp949') as f:
    content = f.read()
content = content.replace('old_영문_코드', 'new_영문_코드')
with open('파일.pas', 'w', encoding='cp949') as f:
    f.write(content)
```

**3단계: 수정 후 반드시 검증**
```bash
python -c "open('파일.pas', encoding='cp949').read(); print('OK')"
# UnicodeDecodeError 발생 시 즉시 git checkout -- 파일.pas 로 복구
```

### 핵심 제약
- `old_string`/`new_string`에 한글 포함 금지 — CP949 바이트 손상 발생
- 한글 주석 추가가 필요하면 Python `content.replace()` 방식만 사용
- 검증 실패 시 `git checkout -- <파일>` 복구 후 재시도

## Delphi 수정 패턴

### 주석 스타일
```pascal
// #이슈번호 변경 이유 (수정 내용 한 줄 요약)
{ #이슈번호 기존 코드 주석 처리:
  OldCode;
}
```

### 프로시저/함수 수정
- 파라미터 추가 시 기존 호출부 전체 확인 필요 (`Grep`으로 함수명 검색)
- `uses` 절에 유닛 추가 시 순환 참조 방지 — 같은 파일에서 서로 참조하지 않는지 확인

### 버전 관리
Delphi 프로젝트는 `{프로젝트명}.dpr`의 버전을 수동으로 관리하지 않음. C# 쪽만 버전 업.

## C# 수정 패턴

### AssemblyInfo.cs 버전 업 (필수)
C# 프로젝트 파일 수정 시 반드시 함께 업데이트:
```csharp
// 수정 전
[assembly: AssemblyVersion("1.0.0.X")]
[assembly: AssemblyFileVersion("1.0.0.X")]

// 수정 후 (마지막 자리 +1)
[assembly: AssemblyVersion("1.0.0.X+1")]
[assembly: AssemblyFileVersion("1.0.0.X+1")]
```

`AssemblyInfo.cs` 위치: 프로젝트 루트 또는 `Properties/` 하위.

### 주석 스타일
```csharp
// #이슈번호 수정 이유 (변경 내용 한 줄 요약)
// #이슈번호 기존 코드: OldCode();
NewCode();
```

### Null 체크
C# 코드 수정 시 null 참조 예외 가능성을 검토한다. YSR은 nullable 어노테이션을 사용하지 않는 구버전 코드가 많다.
