● YSR 유지보수 프로세스

---
기본 사용법

이슈 번호와 버그 현상을 설명하면 자동으로 전체 워크플로우가 실행된다.

  이슈 #207500 확인해줘. 프린터 포트가 갱신 안 된다는 버그야.

  → Phase 1: Redmine 조회 + 코드 탐색 → 01_investigation.md
  → Phase 2: 코드 수정 → 02_patch_summary.md          (확인 후 진행)
  → Phase 3: 커밋 메시지 작성 → 03_commit_message.md

---
부분 실행

  조사만 해줘          → Phase 1만 실행
  재현 방법 알려줘      → Phase 1 실행 (재현 가이드 포함)
  수정까지 해줘        → Phase 1 + 2 실행
  커밋 메시지만 다시 만들어줘  → Phase 3만 재실행 (_workspace/ 재사용)

---
전체 흐름

사용자 요청
   ↓
[Phase 0] _workspace/ 확인
   신규 이슈 → _workspace_prev/로 백업 후 초기화
   재실행 요청 → 해당 Phase만 실행
   ↓
[Phase 1] bug-investigator
   Redmine 이슈 조회 → 코드베이스 탐색 (Common → Module 순)
   → _workspace/01_investigation.md
   ↓
[Phase 2] patch-author                        ← 사용자 확인 후 진행
   CP949 인코딩 규칙 준수 + 최소 범위 수정
   C# 수정 시 AssemblyInfo.cs 버전 업 포함
   → _workspace/02_patch_summary.md
   ↓
[Phase 3] commit-writer
   fix #이슈번호 [카테고리] 제목 형식 + <pre> 블록
   → _workspace/03_commit_message.md
   ↓
커밋 메시지 + git 명령어 사용자에게 전달

---
산출물 적용

_workspace/03_commit_message.md의 안내에 따라:

  git checkout -b 2026_서비스자동화_배포_#이슈번호
  git add 수정된파일들
  git commit -m "fix #이슈번호 ..."
